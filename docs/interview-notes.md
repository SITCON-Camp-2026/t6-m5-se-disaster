# 使用者訪談紀錄

> 這份文件由 Codex 先產生草稿，已保留人類需要再確認的欄位；不要直接把 AI 輸出當成最後結論。

## 我的 Phase 0 prototype 現況

- 目前首頁有「原始資訊」和「整理工作台」兩個 tab。
- 原始資訊保留 12 筆未整理資料，並顯示來源與查核狀態。
- 整理工作台會產生可編輯的 agent 草稿，包含需求分類、候選類型、信心程度、下一步、原文依據、不能直接變任務原因、人工確認備註與安全勾選欄位。
- 草稿會明確標示「不能直接變成志工任務」與「需要人工確認」，避免被誤認為已確認資料。

## Persona 1：回報者

### 我想完成的事

- 我想快速把我知道的狀況送出去，即使地點、數量或需求還不完整。
- 如果我是轉述別人的話，我希望系統保留「不是親眼確認」這件事。
- 我希望後面整理的人知道哪些是我確定看到的，哪些只是群組或他人轉述。

### 我看不懂的地方

- 我不太確定「agent 草稿」和「已整理好的資訊」差在哪裡，看到分類時可能以為系統已經判斷好了。
- 「候選類型」「信心程度」比較像整理者用語，我不知道回報者是否也需要理解。
- 如果原始資訊被送到整理工作台，我不確定它會不會自動變成任務。

### 我最怕的錯誤

- 我只是說「有人說需要幫忙」，結果系統把它變成可以直接派人的任務。
- 我不知道正確地址或數量，但系統幫我補成看起來很確定的地點或需求。
- 我轉述長者、家屬或群組訊息時，系統沒有保留同意公開或未確認的狀態。

### 我不敢直接相信的資訊

- 我不敢相信模糊地點被整理後變成可行動地點，例如「老店旁邊」被理解成明確地址。
- 我不敢直接相信社群貼文或留言裡的更新，因為可能過期、轉述或和現場不一致。
- 我不敢相信「官方公告」這類字樣，除非畫面也說清楚日期、來源與是否已確認。

### 我希望系統提醒我的事

- 如果我填的是轉述，希望系統提醒我標明不是親眼確認。
- 如果我不知道時間、地點、當事人同意或需求是否還存在，希望系統不要逼我猜。
- 如果我的文字可能被後面的人誤解成已確認事實，希望系統提醒我補一句「尚未確認」。

### 我想反問開發者的一個問題

- 如果我只想先留下線索、不想讓它立刻變成任務，這個工作台會不會清楚保護我的不確定描述？

## Persona 2：資訊整理者

### 我想完成的事

- 我想把原始資訊整理到下一位協作者看得懂，但不要把不確定資訊包裝成事實。
- 我想清楚分出原文、agent 推測、人類修正與仍需確認的地方。
- 我想快速找到來源衝突、資訊缺口和不能直接行動的原因。

### 我看不懂的地方

- 我不確定 agent 預填草稿的依據是否完整，尤其是分類和下一步看起來像已經被系統決定。
- 我不清楚「信心程度」是根據什麼判斷，是來源可信、欄位完整，還是 agent 對分類有把握。
- 如果人類修改了草稿，我看不出修改紀錄或誰做了判斷。

### 我最怕的錯誤

- AI 自動補了原文沒有的資訊，例如地點、數量、需求類型或最新狀態。
- 查核狀態、任務狀態和草稿狀態混在一起，讓下一位協作者以為已經確認。
- 沒有留下為什麼不能直接變任務的理由，後面的人只看到分類結果。

### 我不敢直接相信的資訊

- 我不敢直接相信 agent 的需求分類，除非能看到原文依據和不確定點。
- 我不敢直接相信單一來源的現場更新，尤其是社群貼文、留言或轉述。
- 我不敢直接相信已經整理過的文字，除非畫面仍保留原始文字。

### 我希望系統提醒我的事

- 當原文缺少時間、地點、當事人同意或確認來源時，系統要提醒我不要推進成任務。
- 當同一地點或同一需求有衝突資訊時，系統要提醒我保留衝突，不要只選一個看起來合理的版本。
- 當我修改 agent 草稿時，系統要提醒我哪些是我自己的判斷，不是原文事實。

### 我想反問開發者的一個問題

- 這個工作台之後會怎麼記錄「誰在什麼時候根據什麼理由」把草稿推進到下一個狀態？

## Persona 3：行動者

### 我想完成的事

- 我想知道哪些資訊可以行動，哪些只是線索或仍未確認。
- 我想避免根據錯誤、過期或模糊資訊跑錯地方、帶錯物資或做錯事。
- 我想在採取行動前，看見地點、需求、狀態與安全限制是否足夠清楚。

### 我看不懂的地方

- 如果畫面同時顯示「任務候選」「下一步」「不能直接變任務」，我需要很努力才知道到底能不能行動。
- 「信心程度：高」可能讓我以為可以相信，但它不一定代表現場已確認。
- 我不知道目前看到的是最新狀態，還是只是某個時間點的轉述。

### 我最怕的錯誤

- 把未確認資訊當成已確認，結果人去了現場才發現需求不存在或地點錯誤。
- 任務描述和實際狀況不同，造成志工安全風險或浪費現場資源。
- AI 把「可能」「有人說」「疑似」整理成看起來確定的行動指示。

### 我不敢直接相信的資訊

- 我不敢直接相信沒有明確時間的需求，因為現場狀況可能已經變了。
- 我不敢直接相信模糊地點或只靠暱稱描述的地點。
- 我不敢直接相信沒有人工確認紀錄的任務候選。

### 我希望系統提醒我的事

- 如果資訊仍是 `needs_review` 或 `unverified`，請明確提醒我不要直接行動。
- 如果資料可能過期、來源衝突或地點不明，請把風險放在很醒目的地方。
- 如果這筆資料只是整理草稿，請不要用像任務指派或行動指令的語氣呈現。

### 我想反問開發者的一個問題

- 你希望我在看到「任務候選」時做什麼：等待確認、回報補充資訊，還是可以準備但先不要出發？

## 人類補充

- 三個 persona 都會被「分類很完整」這件事誤導，所以分類應該更明確地標成候選。
- 對行動者來說，「不能直接變任務」比需求分類更重要，後續畫面應該把不可行動原因放得更靠前。
- 現階段不應急著加更多自動化，應先把狀態語意、確認責任和人工修改紀錄講清楚。
