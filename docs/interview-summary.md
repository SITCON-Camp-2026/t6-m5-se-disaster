# Main Agent 訪談彙整

> 這份文件由 Codex 先產生草稿，已加入人類確認痕跡區塊；最後決策仍需由人確認。

## 三個 persona 的共同需求

- 使用者明確說的：三個角色都需要分清楚「原始資訊」「agent 草稿」「人工確認後資訊」。
- 使用者明確說的：三個角色都擔心未確認、轉述或模糊資訊被整理成看起來可行動的事實。
- 使用者明確說的：三個角色都需要看到來源、時間、地點清楚度、是否轉述與人工確認狀態。

## 三個 persona 之間的衝突需求

- 使用者明確說的：回報者想快速留下不完整線索；資訊整理者需要足夠脈絡才能降低誤判；行動者需要明確到可以安全判斷是否行動。
- 使用者明確說的：整理者需要較多欄位與判斷紀錄；回報者可能會被太多欄位嚇退。
- 推論：行動者希望畫面直接告訴他能不能行動，但這個判斷不能由 AI 自動完成，必須保留人工確認。

## Phase 0 原型中被推翻的假設

- 原本假設：只要在草稿中標示「需要人工確認」和「不能直接變任務」，使用者就能理解它不是已確認資料。
  - 被什麼回饋推翻：回報者和行動者都可能被「需求分類」「任務候選」「信心程度」這些看起來正式的欄位誤導。
- 原本假設：agent 預填分類能幫助整理速度，而且不會造成太大誤解。
  - 被什麼回饋推翻：資訊整理者擔心分類和下一步看起來像系統已經決定，行動者可能把候選結果當成行動指示。

## 最容易造成誤導的風險

- 使用者明確說的：AI 把「可能」「有人說」「疑似」改寫成看起來確定的內容。
- 使用者明確說的：高信心、任務候選或完整分類讓使用者誤以為資訊已確認。
- 推論：如果原始文字和草稿判斷距離太遠，使用者很難分辨哪些是原文、哪些是 agent 或人類推測。

## 可以考慮的設計方向

> 這裡只是可考慮方向，不是標準答案。

- 把「可否行動」與「需求分類」分開，並讓不可行動原因比分類更醒目。
- 把 agent 草稿標示改得更保守，例如「候選整理，尚未確認」，避免只看到分類結果。
- 在草稿中保留「原文支持」與「需要人確認」兩個明確區塊，讓整理者和行動者能快速判斷風險。
- 對回報者保留低門檻入口，不強迫補完未知欄位，但提醒轉述、時間、地點與同意狀態。

## 需要人類決策的取捨

- 取捨 1：
  - 選項 A：v1 優先服務回報者，降低回報門檻。
  - 選項 B：v1 優先服務資訊整理者，先把判斷流程和狀態責任做清楚。
  - 我目前傾向：選項 B。
  - 理由：Phase 0 已經有整理工作台雛形，而三個 persona 的共同風險都集中在「整理後被誤認為已確認」。先服務整理者比較能降低後續所有角色的誤判。
- 取捨 2：
  - 選項 A：讓行動者看到更多候選任務，方便快速反應。
  - 選項 B：在未人工確認前，不把候選內容包裝成行動任務。
  - 我目前傾向：選項 B。
  - 理由：涉及安全和現場資源，不應讓 AI 或草稿欄位產生行動指示感。

## 不應該讓 AI 自動決定的事情

- 資訊是否已確認、是否可以派工、是否可以公開當事人資訊。
- 衝突來源中哪一個比較可信。
- 模糊地點、需求數量、需求類型或現場最新狀態。
- 志工是否應該出發、帶什麼物資、去哪裡。

## 我採用的 AI 整理

- 採用三個 persona 都在意「未確認資訊被誤解」這個共同風險。
- 採用「原文、agent 推測、人類判斷要分開」作為 v1 設計的核心檢查點。
- 採用「先不要把候選任務做得像行動指令」的保守方向。

## 我不採用的 AI 建議

- 建議：讓 agent 自動根據草稿信心程度推進下一步。
  - 不採用原因：信心程度容易被誤解成查核完成，是否推進必須有人類角色確認。
- 建議：把三個 persona 的需求合併成一份完整功能清單。
  - 不採用原因：這會掩蓋回報速度、整理嚴謹和行動安全之間的衝突。

## 我自己補充的判斷

- 現階段最重要的不是增加功能，而是降低「看起來已完成」的錯覺。
- 「任務候選」這個詞對行動者可能太接近真實任務，後續需要更清楚的視覺和文案區隔。
- 如果 v1 服務資訊整理者，也仍要保留回報者的不完整輸入，不應把欄位要求變得太硬。

## 我仍不確定的地方

- v1 是否要保留「信心程度」，或改成更具體的缺口檢查。
- 誰有權把草稿推進到候選通報或可行動狀態。
- 是否需要在 v1 就做修改紀錄，還是先用簡化的人類確認欄位。
