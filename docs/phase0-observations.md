# Phase 0 Observations

## 我們做出了什麼？

- 首頁已經可以看到 Phase 0 原始資訊列表與整理工作台。
- 整理工作台目前可以在前端記憶體中建立、查看、編輯、刪除與重設整理草稿。
- 草稿只代表候選整理，不代表已確認資料，也不會改動原始資訊檔案。
- 目前沒有串接真實 runtime AI；工作台中的 agent 草稿是 coding agent 協助產生的保守預填內容，仍需人類檢查。
- 工作台新增需求分類、顏色標示與篩選功能，分類包含人力需求、物資需求、專業支援、地點狀態、公告資訊與待判斷。

## 第一階段交付檢查

- 首頁可以看到第一階段原始資訊列表，資料來源是 `src/fixtures/phase-0/messy-reports.json`。
- 整理工作台可以建立、查看、編輯、刪除與重設整理草稿。
- 工作台預設建立 6 筆保守草稿，方便開始整理，但不代表這 6 筆已被確認。
- 目前工作台會替 12 筆原始資訊建立 agent 預填草稿，但這些草稿都保持需要人工確認與不能直接行動。
- 需求分類可以被人類修改，也可以用來篩選草稿；分類本身不是已確認結果。
- 工作台可以把至少 3 筆資訊標示為不能直接變成任務；目前保守預設會先把草稿標成不能直接行動。
- 工作台可以用勾選欄位記錄至少 2 個「人類質疑或修正 agent 判斷」。
- `needs_review` 與 `unverified` 沒有被顯示成已確認。
- 目前沒有新增後端、資料庫、外部 API、地圖、localStorage 或 runtime LLM 呼叫。

## 哪些資料讓 agent 很容易整理？

- 有明確時間、來源描述與限制語句的資料，較容易先整理成候選草稿。
- 例如原文有寫「目前仍開放」「只接受已完成報到」「不再收二手衣物」這類明確限制時，比較容易標出原文依據。
- 即使資訊看起來比較完整，仍需要看查核狀態，不能直接當成 verified。

## 哪些資料讓 agent 亂補或誤判？

- 只有模糊地點的資訊，agent 很容易自行補出地址或可派工地點。
- 只有轉述或留言衝突的資訊，agent 可能把其中一方當成最新事實。
- 寫著「官方公告」來源類型的資料，若原文同時說不知道日期或是否官方，仍不能直接當作官方確認。

## 哪些資訊不能直接變成任務？

- 地點不清楚、時間不清楚或需求是否仍存在不清楚的資訊，不能直接變成任務。
- 操作者不是當事人，或尚未確認當事人同意公開資訊時，不能直接變成任務。
- 來源之間互相衝突，或原文沒有說明原因的「不要派人」資訊，不能直接變成任務。

## 哪些資訊的操作者不是當事人？

- 有志工代長者轉述的資訊，需要確認長者本人意願與可公開範圍。
- 有外地家屬來電轉述親友狀況的資訊，需要確認親友實際位置、需求與同意。
- 社群貼文或群組轉述通常也不是當事人直接回報，需要保留人工確認狀態。

## 哪些資料讓工具看起來有用？

- 同一個地點或物資狀態在不同時間出現不同說法時，工作台可以把原始資訊並排保留，避免只留下單一結論。
- 對於有明確限制的資訊，草稿可以把限制寫進「原文依據」與「不能直接變成任務的原因」。
- 對於轉述型資訊，工作台可以明確標示需要人工確認，避免看起來像可以直接派工。

## 哪些資料讓工具開始不可靠？

- 原文只有模糊地點時，工作台無法靠 UI 補出可安全使用的位置。
- 原文只有「有人說」「留言說」時，工作台無法判斷哪個說法最新或可信。
- 原文沒有說明當事人是否同意公開時，工作台不能自行把資訊轉成任務。

## 我們不敢讓志工直接採取行動的資訊

- 地點模糊、需求模糊或時間不明的資訊，都不應直接派志工。
- 轉述他人需求、且尚未確認當事人同意的資訊，不應直接建立公開任務。
- 有衝突說法的資訊，需要先確認最新狀態，不能只根據其中一段文字行動。

## 我們剛剛偷偷假設了什麼？

- 我們假設「可建立草稿」代表開始整理，而不是代表資訊已經可信。
- 我們假設草稿中的候選類型可以被人類修改，因此不把初始類型寫成標準答案。
- 我們假設所有未確認資料都應保守標示為不能直接行動，除非人類補上確認脈絡。

## 如果要讓這個工具變可靠，下一步需要什麼？

- 需要更清楚的資料格式，分開記錄原文、來源、確認狀態、當事人同意與更新時間。
- 需要人工確認角色來判斷哪些候選草稿可以往下一步走。
- 需要明確狀態規則，避免把「草稿」「候選」「已確認」混在一起。

## 我們需要更清楚資料格式的地方

- 地點描述需要分成原文、可公開程度、是否已確認，不應直接變成地址。
- 需求描述需要記錄是本人需求、轉述需求、還是他人推測。
- 更新時間需要和原文事件時間分開，避免把收到資訊的時間當成現場狀態時間。

## 我們需要人工確認的地方

- 轉述資訊是否取得當事人同意。
- 看似已更新的物資或集合點資訊是否仍有效。
- 衝突資訊應以哪個來源為準，以及是否需要保留多個版本。

## 我們是否需要狀態機？

- 需要，但現在不應在 Phase 0 先做完整狀態機。
- 目前至少已經看出「原始資訊」「整理草稿」「候選結果」「需要人工確認」「已確認」不應混在一起。
- 後續若要擴充，應先定義每個狀態能不能被公開、能不能派工、誰可以把它推進下一步。

## 哪些資料品質問題不是 UI 能解決的？

- 原始資訊本身缺少時間、地點、當事人同意或確認來源時，UI 只能提醒，不能補成事實。
- 多個來源互相衝突時，UI 只能保留衝突與提醒人工確認，不能自動決定誰正確。
- 涉及隱私、公開範圍或安全判斷時，必須有人類角色負責確認。

## 60 秒口頭分享草稿

我們做出了一個 Phase 0 災害資訊整理工作台。首頁可以看到 12 筆原始資訊，也可以進入整理工作台建立與編輯草稿。這個工具最有用的地方，是它不急著把資訊變成任務，而是把來源、查核狀態、不能直接行動的原因、需要人工確認的地方放在同一個畫面。過程中我們發現，agent 很容易把模糊地點、轉述資訊或看起來完整的現場更新整理得太像事實，所以我們保留了「不能直接變成志工任務」和「人類質疑或修正 agent 判斷」欄位。下一步如果要讓 prototype 可靠，需要先定義更清楚的資料格式、人工確認角色和狀態規則。

## 1 個 AI 可能誤導我們的地方

- AI 可能把「原文看起來很完整」誤解成「資訊已確認」，例如直接推論某地點可用、某需求仍存在，或某轉述可以公開派工。這些都必須由人類確認，不能只靠草稿欄位或 agent 判斷。
