# 需求取捨決策

> 這份文件由 Codex 先產生草稿，最後仍需由人類刪除、修改或補充。

## 我決定 v1 優先服務的使用者

- [ ] 回報者
- [x] 資訊整理者
- [ ] 行動者

原因：

Phase 0 原型已經證明工作台可以快速產生候選整理，但訪談回饋顯示最大風險是候選整理被誤認為已確認資料。v1 先服務資訊整理者，可以優先釐清原文、agent 推測、人類判斷、查核狀態和不可行動原因，降低後續回報者與行動者被誤導的機率。

## 這個使用者最想完成什麼

- 快速看懂大量原始資訊，但不把未確認內容包裝成事實。
- 找出資訊缺口、來源衝突、轉述風險和不能直接變任務的原因。
- 留下判斷依據，讓下一位協作者知道哪些是原文、哪些是人類判斷或 agent 草稿。

## 這個使用者最怕什麼錯誤

- AI 自動補完原文沒有的地點、數量、需求類型或最新狀態。
- 查核狀態、任務狀態和草稿狀態混在一起。
- 下一位協作者只看到分類結果，看不到為什麼不能直接相信或行動。

## 我今天暫時不處理的需求

- 暫時不做給行動者直接使用的任務派工或出發建議。
- 暫時不做自動合併重複資訊或自動判斷衝突來源可信度。
- 暫時不要求回報者填完整表單，避免降低回報意願。

## 我不採用的 AI 建議

- AI 建議：根據 agent 草稿的信心程度自動建立候選通報或任務。
  - 不採用原因：信心程度不是查核狀態，也不能代表現場已確認；推進狀態必須保留人工確認。
- AI 建議：把三個 persona 的需求整合成完整 v1 功能清單。
  - 不採用原因：三個角色有明顯衝突，直接合併會讓範圍過大，也會掩蓋安全取捨。

## 必須保留人工確認的判斷

- 是否可以把原始資訊推進為候選通報或任務。
- 轉述資訊是否取得當事人同意，以及是否適合公開。
- 衝突資訊中哪個版本可採用，或是否應全部保留等待確認。
- 地點、需求、時間和狀態是否足夠讓後續協作者安全使用。

## 今天明確不做什麼

- 不讓 AI 自動決定資訊已確認。
- 不把候選任務呈現成可以直接派工或出發。
- 不新增後端、資料庫、外部 API、地圖或 runtime LLM。
- 不補真實地址、電話、人物或現場資料。

## 我仍不確定的地方

- v1 要不要保留「信心程度」欄位，或改成更明確的缺口檢查清單。
- 修改紀錄是否要在 v1 實作，還是先用人工確認備註替代。
- 回報者和資訊整理者之間的交接格式需要多正式才夠用。
