# Main Agent 訪談彙整

> 這份文件整理三個 persona 回饋，以及後續實作後的產品取捨。最後決策仍需由人類確認。

## 三個 persona 的共同需求

- 使用者明確說的：所有角色都需要清楚分辨「原始資訊」「系統預填草稿」「人工確認後資訊」。
- 使用者明確說的：大家都擔心未確認、轉述或模糊資訊被整理得太像事實，進而讓人誤以為可以直接行動。
- 使用者明確說的：來源、時間、地點、是否轉述、是否需要人工確認，都必須在畫面上被看見。
- 推論：回報、整理、行動雖然是不同工作，但可以在同一平台中串起來，只要狀態和責任不要混在一起。

## 三個 persona 之間的衝突需求

- 回報者想快速留下線索，不想被完整表單卡住。
- 資訊整理者需要更多欄位、分類、原文依據和人工確認備註，才能避免誤判。
- 行動者希望知道「能不能接、能不能做、完成多少」，但這不能被誤解成真實救災派工。
- 最後採用的折衷是：把回報者和行動者整合成「回報與行動者」，讓同一個人可以留下線索、接 demo 任務、完成 demo 任務；資訊整理者則負責看分類、確認狀態和排行榜。

## Phase 0 原型中被推翻的假設

- 原本假設：只要標示「需要人工確認」和「不能直接變任務」，使用者就不會誤解。
  - 被推翻原因：分類、任務候選、接單、完成數這些 UI 都很容易讓人以為事情已經被確認，所以必須反覆標示「這只是 demo / 未確認」。
- 原本假設：回報者、行動者、整理者要完全分開。
  - 被推翻原因：回報者和行動者在這個 prototype 中很接近，都需要面對未確認資訊；合併後流程更簡單，也比較像平台上的使用者端。
- 原本假設：排行榜只需要資訊整理者看到。
  - 被推翻原因：行動者也需要即時看到自己的完成數，排行榜應同時出現在回報與行動者畫面和資訊整理者畫面。

## 最容易造成誤導的風險

- 把「接單」誤解成真實派工。
- 把「完成任務」誤解成現場狀態已解決。
- 把「API 預填分類」誤解成 AI 已經確認資訊。
- 把排行榜當成鼓勵速度的競賽，導致使用者忽略安全與人工確認。
- 把留言數當成可信度，而不是互動或補充資訊。

## 可以考慮的設計方向

- 回報與行動者畫面保留低門檻回報，同時顯示接單區、完成按鈕與排行榜。
- 資訊整理者畫面顯示分類總覽、可行動狀態、預填 API 狀態和任務完成排行榜。
- 原始資訊頁提供新增資訊與留言，但所有新增內容都預設為 `needs_review`。
- 接單區可以像 foodpanda 平台一樣呈現「可接單」「已接單」「已完成」，但文案要持續提醒它只是 demo 任務。
- 分類總覽做成可折疊，避免整理者在掃描資料時被統計資訊佔滿畫面。
- 登入加入 CAPTCHA demo 流程，讓不同角色進入前先經過後端挑戰；但測試答案顯示在畫面上，只能代表流程，不代表真實安全。
- 資料異動統一走本地 backend service，讓前端狀態更新更像產品架構；但目前仍不是獨立 server 或資料庫。

## 需要人類決策的取捨

- 取捨 1：
  - 選項 A：把接單做得像真實派工平台。
  - 選項 B：只做 demo 接單，明確保留未確認提醒。
  - 我目前傾向：選項 B。
  - 理由：這是災害資訊原型，不應讓使用者誤以為可以根據未確認資料直接出發。
- 取捨 2：
  - 選項 A：回報者、行動者、資訊整理者三種角色分開。
  - 選項 B：整合回報者與行動者，資訊整理者獨立。
  - 我目前傾向：選項 B。
  - 理由：回報與行動者都接觸原始線索和安全提醒，合併後流程較清楚；資訊整理者仍需要獨立工作台。
- 取捨 3：
  - 選項 A：排行榜鼓勵完成數。
  - 選項 B：排行榜只作為 demo 指標，不能取代品質與確認。
  - 我目前傾向：選項 B。
  - 理由：救災情境不能只追求完成數，仍要保留人工確認與安全判斷。

## 不應該讓 AI 自動決定的事情

- 資訊是否已確認。
- 是否可以真實派工或出發。
- 衝突來源中哪個版本可信。
- 模糊地點、需求數量、需求類型或現場最新狀態。
- 任務完成是否代表現場問題已解決。
- 排行榜是否代表使用者表現品質。

## 我採用的 AI 整理

- 採用「未確認資訊不能包裝成事實」作為所有畫面的核心限制。
- 採用「資訊整理者優先」作為主要工作台方向。
- 採用「回報者與行動者合併」來簡化角色流程。
- 採用「接單與排行榜」來模擬平台關係，但保留 demo 與安全提醒。

## 我不採用的 AI 建議

- 建議：讓 API 預填結果直接成為正式分類。
  - 不採用原因：API 預填只能是草稿，仍需要人工確認。
- 建議：接單後就視為可行動任務。
  - 不採用原因：接單只是前端 demo 狀態，不能代表現場資訊已確認。
- 建議：排行榜直接評價使用者能力。
  - 不採用原因：完成數只是一個操作紀錄，不代表任務品質、資訊可信度或安全程度。

## 我自己補充的判斷

- 現在的 prototype 比較像「災害資訊平台雛形」：有使用者角色、資料新增、互動、分類、接單、完成數和排行榜。
- 最大風險也因此變高：越像平台，越容易讓人以為資料真的可以行動。
- 所以每個新功能都應該配一個安全限制：新增資訊預設未確認、API 預填是草稿、接單不是派工、完成不是現場解決、排行榜不是品質評分。

## 我仍不確定的地方

- 是否需要把「完成任務」改名成更保守的「回報已處理」或「demo 完成」。
- 排行榜是否應顯示更多品質指標，例如「被退回數」「待確認數」「有留言補充數」。
- 接單是否應只允許已人工確認的資料出現，還是保留目前 demo 方式。
- 回報與行動者是否需要不同模式切換，例如「我要回報」和「我要接單」。
- 下一版是否要把本地 backend service 換成真正 API server、資料庫和正式 CAPTCHA provider。
